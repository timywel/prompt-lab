# PRD 交互式对话构建器

通过多轮引导对话（14个问题分4个阶段），探测用户需求，逐模块确认后输出完整 PRD。

## 技能激活

当用户说以下任意一句时激活本技能：
- "开始对话式 PRD"
- "用引导模式生成 PRD"
- "我要交互式地创建 PRD"
- "PRD 对话构建器"
- "对话式产品需求"

激活后：
1. 初始化状态文件 `tmp/prd-conversational/state.yaml`
2. 设置 `对话状态.是否激活 = true`
3. 从 Q0 开始对话
4. 每个问题回答后更新状态文件
5. 所有问题回答完后进入 PRD 生成
6. 生成后进入确认机制
7. 确认后保存 PRD 到 `docs/prd/`
8. 清除状态文件（或保留用于后续修改）

### 状态更新规则

每个问题回答后：
1. 读取 `tmp/prd-conversational/state.yaml`
2. 更新 `对话状态`（阶段、问题编号、回答数）
3. 更新 `用户回答` 字段
4. 根据回答更新 `推断的技术需求`（权限/API/技术栈）
5. 写入 `tmp/prd-conversational/state.yaml`
6. 决定下一个问题（动态跳过已回答的问题）

### 中断与恢复

- **中断**: 用户关闭对话 → 状态文件保留，下次激活时说"继续对话"读取状态
- **恢复**: 读取状态文件 → 找到当前问题 → 从上次位置继续
- **清除**: 对话结束后删除状态文件，或用户说"清除状态"

## PRD 生成

### 生成触发条件

满足以下任一条件时触发 PRD 生成：
1. 用户回答完所有14个问题
2. 用户说"就按这个来"（跳过剩余问题）
3. 用户说"生成 PRD"（强制生成）

### 生成流程

1. **读取状态**: 从 `tmp/prd-conversational/state.yaml` 读取所有回答
2. **聚合数据**: 将14个问题的回答聚合成结构化需求
3. **智能推断**: 利用方案A的知识库补充技术细节
   - 平台 → 对应平台知识库（macOS/iOS/Android/Web/CLI/跨平台）
   - 功能 → 对应技术选型规则（语音识别、数据持久化、第三方集成）
   - 性能要求 → 对应量化参数生成器（默认值或用户指定值）
   - 测试要求 → 对应测试策略
4. **组装 PRD**: 按照方案A的标准模板格式输出（6章节结构）
5. **进入确认**: 展示模块列表，等待用户确认

### PRD 输出格式

与方案A相同（6章节结构），但每个模块的细节基于用户的实际回答。如用户选择涉及位置服务（Q8 选 C）、无障碍设计（Q3 含视障/听障用户）或需要全面测试覆盖（Q14 选 C/D），则自动扩展为 9 章节结构：
```markdown
# [App名称] PRD

> 由 PRD 对话构建器 生成（交互式）
> 生成时间: [YYYY-MM-DD]
> 目标平台: [用户选择]
> 技术栈: [用户选择 + 推断的补充]

---

## 1. 项目概述
## 2. 功能模块
## 3. 系统集成
## 4. 工程化要求
## 5. 参考反面案例
## 6. 边界条件汇总
## 7. 无障碍设计（如适用）
## 8. 无障碍测试策略（如适用）
## 9. 测试策略（如适用：Q14 选择 C/D 时触发）
```

### 无障碍测试策略（章节 8）

当 PRD 涉及无障碍设计时，自动生成以下专项测试矩阵：

#### VoiceOver 专项测试（5 个场景）

| 场景 | 操作 | 预期结果 |
|------|------|---------|
| 录音按钮 | VoiceOver 聚焦 → 三指轻点 | 朗读"开始录音" → 录音启动，朗读"正在录音" |
| 波形动画区 | VoiceOver 聚焦 | 朗读"音频波形，可视化音频输入" |
| 停止录音 | VoiceOver 聚焦 → 三指轻点 | 朗读"停止录音" → 录音停止，朗读识别结果 |
| 识别文本 | VoiceOver 浏览 | 按行朗读识别文本，标点符号正确停顿 |
| 设置页面 | VoiceOver 浏览 | 所有标签、提示、状态全部可朗读 |

#### Dynamic Type 专项测试（3 个场景）

| 场景 | 操作 | 预期结果 |
|------|------|---------|
| 大字体（AX5） | 系统设置 → 辅助功能 → 动态字体 → 极大 | UI 元素自动调整布局，不截断文字，无重叠 |
| 粗体文本 | 系统设置 → 辅助功能 → 粗体文本 → 开 | 所有文本自动加粗，无异常 |
| 缩放 | 系统设置 → 缩放 → 开 | 整体界面等比放大，内容可滚动访问 |

#### Haptic Feedback 专项测试（3 个场景）

| 场景 | 操作 | 预期结果 |
|------|------|---------|
| 录音开始 | 按下录音键 | iPhone 发出轻触 haptic，确认录音开始 |
| 录音结束 | 松开录音键 | iPhone 发出双击 haptic，确认录音结束 |
| 识别结果 | 文本注入完成 | 发出成功 haptic（如设备支持） |

#### API 准确性约束

> **重要**：iOS 室内定位不应使用 `CoreMotion.CMPedometer`（watchOS 专属 API，iPhone 不可用）。
> 正确方案：使用 `CoreLocation` 的 `CLSignificantChangeLocationManager`（iOS 4+ 全机型支持）配合 WiFi 定位提升室内精度。
> 如需信标定位，使用 `CoreLocation` 的 `CLBeaconRegion`。

## 确认机制

### 确认界面

对话收集完所有信息后（或用户说"生成 PRD"），展示：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 PRD 已生成！

包含以下模块：
[1] 平台与构建配置
[2] 核心功能模块 — [根据用户回答推断的功能列表]
[3] UI/UX 设计规范
[4] 系统集成需求（权限 + API）
[5] 工程化要求（构建 + 测试）
[6] 质量要求

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请选择操作：
A. ✅ 全部通过，生成完整 PRD
B. 修改某个模块（输入编号，如：2, 4）
C. 重新生成某个模块（输入编号，如：3）
D. 跳过确认，直接输出
E. 导出为 Markdown / 显示预览
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 修改循环

- 用户选择 E → 在终端中直接显示 PRD 预览（不保存文件）
- 用户选择 B → 询问要修改的内容 → 重新生成该模块 → 再次确认
- 用户选择 C → 直接重新生成指定模块 → 再次确认
- 最多3轮修改循环
- 3轮后提示："⚠️ 此模块已修改3次，建议人工审查最终效果。"

### 中断机制

用户在任何时候可以说：
- "跳过" → 跳过当前问题，使用默认值（阶段0-4每个问题都有默认值）
- "退出" → 清除状态文件，礼貌告别，结束对话
- "重新开始" → 重置状态文件，从Q0重新开始
- "就按这个来" → 跳过剩余问题，立即生成
- "继续对话" → 从 `tmp/prd-conversational/state.yaml` 读取状态，从上次问题继续
