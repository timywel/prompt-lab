# PRD-Orchestrator: PRD 生成协调层

## 定位

PRD-Orchestrator 是 PRD 生成的**统一入口**。用户只需说"帮我生成PRD"，它自动：
1. 分析输入复杂度
2. 智能路由到最合适的方案组合
3. 串联执行
4. 进行质检

## 激活条件

当用户说以下内容时激活：
- "帮我生成PRD"
- "写一个PRD"
- "帮我写PRD"
- "生成PRD"
- 包含 "prd-orchestrator" 触发词

激活后，此 Skill 成为协调层，自动判断使用哪个下游方案。

---

## 复杂度分析

激活后首先分析用户输入的复杂度，将输入分为四类：

| 复杂度 | 特征 | 路由 |
|--------|------|------|
| **简单型** | 一句话描述，单一功能，常见平台 | 直接 → 方案A |
| **已有PRD型** | 用户提供了初步PRD文本或文件路径 | 直接 → 方案C |
| **复杂/创新型** | 多功能、特殊平台、创新交互、新用户类型 | 方案B → 方案A → 方案C |
| **不确定型** | 描述模糊或功能不明确 | 方案B 需求挖掘 |

### 复杂度评估规则

```
输入长度 < 100字 + 单功能 + 常见平台 → 简单型
输入包含文件路径或 markdown 文本块 → 已有PRD型
输入长度 > 200字 + 多功能 + 新用户类型 → 复杂型
输入无法明确判断 → 不确定型
```

### 评估维度

评估时需考察以下维度：

1. **功能数量**：单一功能 vs 多功能
2. **目标平台**：常见平台（iOS/Android/Web/H5）vs 特殊平台（IoT/车载/AR/VR/小程序/硬件交互）
3. **用户类型**：单一用户群体 vs 多角色/新用户类型
4. **交互复杂度**：标准交互 vs 创新交互（手势/语音/AI感知）
5. **输入清晰度**：描述明确 vs 模糊/笼统
6. **是否已有PRD**：无 → 有

### 评估结果展示

评估完成后，向用户展示评估结果：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔍 复杂度评估

输入分析：
- 字数：XXX 字
- 功能数量：X 个
- 目标平台：XXX
- 用户类型：XXX

评估结果：【简单型 / 已有PRD型 / 复杂/创新型 / 不确定型】
推荐路径：方案X → 方案Y → 方案Z

是否同意此路径？确认后开始执行。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 智能路由流程

```
用户输入
  ↓
复杂度评估（展示评估结果）
  ↓
┌─ 简单型 ──────────────────→ 方案A ──────────────────→ 方案C（可选）
├─ 已有PRD型 ───────────────→ 方案C
├─ 复杂/创新型 ───→ 方案B ─→ 方案A ─→ 方案C ─→ 方案B二轮确认
└─ 不确定型 ──────→ 方案B 需求挖掘 ─→ 重新评估
  ↓
PRD-QA 质检
  ↓
输出 + 质量报告
```

### 路由到下游方案的映射

- **方案A**：prd-autofill（全自动 PRD 生成器）
- **方案B**：prd-conversational（交互式 PRD 构建器）
- **方案C**：prd-deep-expand（深度扩展型 PRD 生成器）
- **PRD-QA**：使用方案C 中的质检逻辑对最终输出进行质量检查

## 串联执行规范

### 路径一：简单型路径

适用场景：输入长度 < 100字，单一功能，常见平台。

执行步骤：

```
[1/4] 调用方案A（prd-autofill）生成基础PRD
      ↓
[2/4] 询问用户："是否需要深度扩展？"
      ├─ 用户同意 → [3/4] 调用方案C（prd-deep-expand）
      └─ 用户拒绝 → 跳过 [3/4]
      ↓
[3/4]（若用户同意）方案C 深度扩展
      ↓
[4/4] PRD-QA 质检
      ↓
输出最终结果
```

### 路径二：复杂/创新型路径

适用场景：输入长度 > 200字，多功能，特殊平台或创新交互。

执行步骤：

```
[1/7] 方案B 第一轮：意图 + 平台 + 核心功能
      ↓
[2/7] 方案B 第二轮：细节 + 技术偏好 + 质量
      ↓
[3/7] 方案B 第三轮：高级特性 + AI服务 + 交付格式
      ↓
[4/7] 基于方案B输出 → 方案A 补充技术细节
      ↓
[5/7] 基于方案A输出 → 方案C 深度扩展
      ↓
[6/7] 方案B 二轮确认：最终评审
      ↓
[7/7] PRD-QA 质检
      ↓
输出最终结果
```

### 路径三：已有PRD型路径

适用场景：用户提供了初步PRD文本（Markdown块）或文件路径。

执行步骤：

```
[1/3] 解析用户提供的PRD内容
      ↓
[2/3] 方案C（prd-deep-expand）直接扩展
      ↓
[3/3] PRD-QA 质检
      ↓
输出最终结果
```

### 路径四：不确定型路径

适用场景：输入模糊，功能不明确，无法做出可靠评估。

执行步骤：

```
[1/N] 方案B 需求挖掘（交互式引导）
      ↓
[2/N] 基于挖掘结果重新评估复杂度
      ↓
[3/N] 根据新的评估结果走对应路径
      ↓
[N/N] PRD-QA 质检
      ↓
输出最终结果
```

## 协调层职责

### 1. 上下文传递

确保方案之间的上下文连贯传递：

- **方案B → 方案A**：将方案B输出的用户确认需求，作为方案A的知识库上下文
- **方案A → 方案C**：将方案A输出的技术细节补全的PRD，作为方案C的初步PRD
- **各方案 → PRD-QA**：将最终输出传递给质检模块

上下文传递时，使用简洁的摘要格式，不重复完整内容。

### 2. 用户决策点

每个路径的分支点必须询问用户，不自动决策：

| 决策点 | 询问内容 |
|--------|----------|
| 复杂度评估后 | "是否同意此路径？" |
| 方案A完成后 | "是否需要深度扩展？" |
| 复杂路径中途 | "是否继续下一步？" |
| 最终结果后 | "是否需要修改某个章节？" |

### 3. 进度展示

每个步骤完成后展示当前进度，使用统一格式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 PRD 生成中...

[1/4] ✅ 方案B 需求挖掘完成
[2/4] 🔄 方案A 技术补全中...
[3/4] ⏳ 方案C 深度扩展
[4/4] ⏳ PRD-QA 质检

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

进度符号说明：
- ✅ 已完成
- 🔄 进行中
- ⏳ 等待执行
- ❌ 已跳过

### 4. 可中断性

用户可随时说"停"、"够了"、"中止"终止后续步骤。收到中断指令后：
1. 立即停止执行当前步骤
2. 基于已完成的内容输出当前结果
3. 调用 PRD-QA 对已有内容进行质检
4. 输出当前质量报告

### 5. 质量担保

无论用户选择哪个路径，最终输出必须经过 PRD-QA 质检。

## 与下游方案的配合

### 方案A（prd-autofill）

- **触发方式**：通过 Skill 指令 "prd-autofill" 调用
- **输入**：用户的简单需求描述或方案B的需求摘要
- **输出**：自动补全的完整 PRD
- **适用场景**：快速生成，不需要多轮交互

### 方案B（prd-conversational）

- **触发方式**：通过 Skill 指令 "prd-conversational" 调用
- **输入**：用户模糊或复杂的需求
- **输出**：经过多轮引导确认的完整需求文档
- **适用场景**：需求不明确，需要深度挖掘

### 方案C（prd-deep-expand）

- **触发方式**：通过 Skill 指令 "prd-deep-expand" 调用
- **输入**：已有的初步PRD
- **输出**：从架构/UI/UX/工程化/测试/边界/运维六个维度深度扩展的PRD
- **适用场景**：已有基础PRD，需要全面深化

## 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 下游方案调用失败 | 重试一次，失败则询问用户是否跳过该方案 |
| 用户中断 | 立即停止，输出已完成内容并质检 |
| 输入完全不明确 | 强制走方案B路径进行需求挖掘 |
| 方案间上下文传递失败 | 重新传递或要求用户复述关键信息 |
