## 2. 功能模块

### 2.1 卡牌系统

**描述**: 以三国武将为主题的扑克牌系统，包含斗地主的标准54张牌（黑桃/红心/梅花/方块 A-K + 大小王），每张牌面绘制像素风格武将立绘。

**技术实现**:
- **核心 API**: PixiJS `Sprite` / `AnimatedSprite` — 像素图渲染
- **输入/触发**: 游戏发牌阶段自动触发，每张牌生成一个 `CardSprite` 实例
- **处理流程**:
  1. 加载卡牌 sprite sheet（单张 64×64px，4bit 色深，像素对齐禁用抗锯齿）
  2. 解析牌面值（花色 + 点数）到纹理坐标映射
  3. 逐张创建 `PIXI.Sprite` 并设置 `pixelPerfect = true`
  4. 应用像素缩放（整数倍放大，如 4x 8x）保持像素锐利
- **输出结果**: 牌桌上手牌/底牌/出牌的视觉展示

**量化参数**:
- 卡牌 sprite 尺寸: 64×64px（1x），显示 256×256px（4x 放大）
- 缩放层级: 1x / 2x / 4x / 8x（根据视口宽度自动选择）
- 发牌动画: 每张间隔 150ms，弹道从牌堆飞向手牌位置
- 帧率: 60fps（PixiJS ticker 锁定）

**UI/UX 规范**:
- 窗口尺寸: 响应式，最大 1200×800px，棋牌桌居中
- 背景: 像素风格木纹/玉玺纹桌布纹理（512×512 平铺）
- 选中效果: 卡牌上浮 16px + 金边高亮（#FFD700）闪烁
- 字体: 自定义像素字体（Press Start 2P / Noto Sans SC 像素渲染）

**反面案例**:
- ❌ **不要**对像素图启用双线性过滤（会导致模糊）— 必须设置 ` PIXI.settings.SCALE_MODE = PIXI.SCALE_MODES.NEAREST`
- ❌ **不要**用 PNG JPG 压缩（有损，边缘伪影）— 必须使用无损 PNG 或 WebP

**边界条件**:
- 断网重连: 缓存当前牌局状态到 localStorage，恢复时重建场景
- 低性能设备: 检测 `navigator.hardwareConcurrency < 4` 时降级为 Canvas2D 模式

### 2.2 斗地主规则引擎

**描述**: 完整实现斗地主规则，包括发牌、叫地主、出牌（含单张/对子/三张/三带一/三带二/顺子/连对/飞机/炸弹/王炸）、胜负判定。

**技术实现**:
- **核心 API**: 纯 TypeScript 类 `LandlordRuleEngine`
- **输入/触发**: 用户点击"出牌"按钮时调用
- **处理流程**:
  1. 验证所选牌型（调用 `RuleEngine.classify(cards)`）
  2. 与上家出牌比较（调用 `RuleEngine.canBeat(selected, lastPlayed)`）
  3. 合法：广播出牌事件；不合法：弹出提示并禁止提交
  4. 无人出牌：当前出牌者获自由出牌权
  5. 胜负判定：`players[i].cards.length === 0` → 立即结算
- **输出结果**: 出牌是否合法、系统强制不出（过）、胜负结算

**量化参数**:
- 规则判定耗时: < 5ms（纯计算，无 IO）
- 出牌响应: < 16ms（1帧内完成判定）

**UI/UX 规范**:
- 出牌按钮: 像素风格木质按钮 120×48px，圆角 0px（像素锐边）
- 提示高亮: 合法最小出牌组合用蓝色描边闪烁
- 不合法反馈: 按钮抖动 + 红色警告文字"请选择正确的牌型"

**反面案例**:
- ❌ **不要**在规则引擎中使用浮点数（精度问题）— 牌面值用整数枚举
- ❌ **不要**硬编码判断（如 `if (type === 'three')`）— 必须用牌型类继承体系

**边界条件**:
- 同时出完: 地主 vs 农民先手判定（按出牌顺序）
- 癞子/变牌规则: MVP 版本不包含，预留 `JokerRule` 接口

### 2.3 AI 对战系统

**描述**: 内置 AI 对手（两个农民 AI），实现叫地主评估、出牌策略（主动/防守）。

**技术实现**:
- **核心 API**: `AIController` 类 + 策略模式 `IStrategy`
- **输入/触发**: AI 回合开始时触发（`turnStart` 事件）
- **处理流程**:
  1. AI 思考延迟 800-1500ms（随机，避免机械感）
  2. 根据策略计算最优出牌（叫地主用概率加权，出牌用 minimax 简化版）
  3. 动画展示 AI 选牌
  4. 提交出牌到规则引擎
- **输出结果**: AI 选择的手牌 + 出牌动画

**量化参数**:
- AI 决策延迟: 800-1500ms（随机均匀分布）
- 策略计算耗时: < 50ms（限制搜索深度 3 层）

**AI 策略设计**:

| 场景 | 策略 | 说明 |
|------|------|------|
| 叫地主 | 概率加权 | 手牌强度 > 阈值（根据牌力模型评分 > 12）|
| 主动出牌 | 优先出短顺/对子 | 快速跑牌 |
| 配合出牌 | 农民间传递信号 | 出对家缺的牌型 |
| 防守出牌 | 压制地主 | 优先打地主报单/报双 |

**反面案例**:
- ❌ **不要**让 AI 每回合固定延迟相同时间（机械感严重）— 必须加抖动
- ❌ **不要**让 AI 看到所有手牌后最优决策（作弊）— AI 只能看到自己的牌

**边界条件**:
- AI 超时: 强制选择"不出"（过）
- 性能不足: 降低 AI 搜索深度（`MAX_DEPTH: 3 → 2`）

### 2.4 像素美术资源

**描述**: 所有视觉资产均为像素风格，4bit 色深（16 色）为主。

**技术实现**:
- **核心 API**: PixiJS `Sprite` + 自定义 shader 抖动
- **输入/触发**: 资源加载阶段
- **处理流程**:
  1. 加载 sprite sheet（合并所有卡牌、小部件到单图）
  2. 生成 @1x @2x @4x @8x 多分辨率 sprite sheet
  3. 根据设备像素比（DPR）选择资源集
  4. 禁用各向异性过滤 + mipmap
- **输出结果**: 场景中所有像素元素的纹理资源

**像素调色板**（斗地主主题）:

| 主题色 | 色值 | 用途 |
|--------|------|------|
| 魏红 | #C41E3A | 黑色花色/魏势力卡片 |
| 蜀绿 | #228B22 | 梅花/蜀势力卡片 |
| 吴蓝 | #1E90FF | 方块/吴势力卡片 |
| 中立金 | #DAA520 | 大王/将卡片框 |
| 暗金 | #8B6914 | 小王/卡背 |
| 木桌色 | #8B4513 | 背景桌布 |
| 玉石色 | #40E0D0 | 按钮高亮 |

**反面案例**:
- ❌ **不要**在像素图上叠加 CSS 模糊效果 — 会破坏像素感
- ❌ **不要**跨分辨率使用同一 sprite sheet（小屏强制缩放会模糊）

**边界条件**:
- WebGL 不可用: 降级到 Canvas2D + 像素缩放
- 极小视口（< 320px）: 横屏提示"请横屏体验"

### 2.5 音效与反馈

**描述**: 像素风格的音效（8-bit 芯片音乐风），卡牌操作触感反馈。

**技术实现**:
- **核心 API**: Web Audio API `AudioContext` + 单音轨音效
- **输入/触发**: 卡牌操作、胜负结算等事件触发
- **处理流程**:
  1. 预加载所有音效到 `AudioBuffer`
  2. 用户首次交互后激活 AudioContext（浏览器策略）
  3. 事件触发时播放对应音效（支持叠加）
- **输出结果**: 音效播放（卡牌翻转/出牌/胜利/失败）

**量化参数**:
- 音效延迟: < 10ms（AudioContext 调度精度）
- 同屏音效叠加上限: 4 个（防止爆音）

**反面案例**:
- ❌ **不要**直接使用 `new Audio()`（延迟高）— 必须用 Web Audio API
- ❌ **不要**自动播放音效（浏览器拦截）— 必须用户交互后激活

## 6. 边界条件汇总
