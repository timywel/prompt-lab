# 方案A设计文档：全自动补全型 PRD 生成器

## 状态
已接受

## 上下文
需要设计一个 PRD 生成器 agent/skill，用户只需输入一句话想法（如"做一个macOS语音输入App"），系统自动补全所有技术细节，输出可直接投喂给大模型执行的完整 PRD 文档。

## 目标
让大模型能生成像参考示例那样具体、详细、具备可执行性的 PRD。

---

## 1. 核心设计理念

### 1.1 参考样例的 PRD 结构分析

参考示例（macOS 语音输入 App prompt）具有以下特征：

| 特征维度 | 具体表现 |
|---------|---------|
| **平台精确** | macOS 14+, Swift, 具体API名称(NSPanel, CGEvent, NSVisualEffectView) |
| **量化指标** | 56px高度, 28px圆角, 44×32px波形区域, 160-560px弹性宽度 |
| **动画参数** | 攻击40%/释放15%, ±4%随机抖动, 0.35s弹簧动画 |
| **反面案例** | "no hardcoded fake animations", "prevent CJK IM from intercepting Cmd+V" |
| **边界条件** | CJK输入法的特殊处理、LLM纠错的保守策略 |
| **工程约束** | LSUIElement模式、SPM构建、Makefile、签名bundle |
| **用户意图推断** | 从"Fn键录音"推断出全局热键+权限+emoji picker抑制需求 |

### 1.2 核心原则

1. **零假设输入**：用户说什么就是什么，但必须自动补全没说清楚的部分
2. **技术深度匹配**：桌面App的PRD比Web App需要更细节的API描述
3. **量化优先**：所有尺寸、时间、参数都必须有具体数值
4. **保守纠错**：LLM的PRD不应过度设计，只补充必要的技术实现路径

## 2. 系统架构

### 2.1 整体架构

```
用户输入（一句话想法）
    ↓
意图识别层（Intent Classifier）
    ↓
平台检测 + 技术栈推断
    ↓
规范补全引擎（Specification Engine）
    ├── 平台知识库（Platform KB）
    ├── 技术选型规则库（Tech Rules）
    ├── 量化参数生成器（Quant Generator）
    └── 反面案例库（Anti-Patterns）
    ↓
PRD 聚合器
    ↓
结构化 PRD 文档（可直接投喂给大模型）
```

### 2.2 各层职责

#### 意图识别层
- 解析用户输入，提取：
  - **目标平台**：macOS/iOS/Android/Web/Backend/跨平台
  - **核心功能**：语音/图像/网络/数据持久化/系统集成
  - **交互模式**：键盘/触摸/语音/手势/全局快捷键
  - **约束条件**：性能要求、兼容性要求、第三方依赖

#### 规范补全引擎

**平台知识库**：
每个平台预定义以下知识：
- 推荐的开发语言/框架
- 必需的权限/ Entitlements
- 常见系统集成API
- UI 框架选择
- 构建工具链
- App Store / 分发要求

**技术选型规则库**：
- 输入方式 → 推荐API（如：语音 → Speech Framework / Whisper / Vosk）
- 持久化需求 → 推荐方案（如：简单配置 → UserDefaults, 复杂数据 → SQLite/CoreData）
- 网络需求 → 推荐方案（REST API → URLSession, WebSocket → Starscream等）
- 跨平台 → Flutter/React Native/Electron的具体对比

**量化参数生成器**：
为每类功能模块生成量化指标：
- 性能指标：启动时间、响应延迟、内存占用、CPU占用、耗电量
- UI参数：常见尺寸、间距、圆角、颜色
- 动画参数：标准时长、缓动曲线、帧率要求
- 兼容性：最低版本要求、硬件要求、屏幕尺寸适配

**反面案例库**：
针对每类常见错误，生成"避坑指南"：
- "不要使用hardcoded的假数据"
- "不要假设网络总是可用的"
- "不要在主线程执行耗时操作"
- 平台特定反面案例（如iOS不要用私有API）

## 3. PRD 输出格式

### 3.1 标准 PRD 模板

生成的 PRD 必须包含以下章节：

```markdown
# [App名称] PRD

## 1. 项目概述
- 项目类型: [macOS桌面App / iOS App / Web应用 / 等]
- 目标平台: [具体版本要求]
- 核心功能: [一句话描述]
- 技术栈: [语言 + 主要框架]
- 构建工具: [SwiftPM / Gradle / npm / 等]

## 2. 功能模块（编号列表）

### 2.[N] [功能名称]
**描述**: [功能的具体描述]

**技术实现**:
- 核心API: [具体API名称，如 Speech Framework]
- 输入/触发条件: [具体的触发方式]
- 处理逻辑: [详细的处理步骤]
- 输出结果: [具体的输出形式]

**量化参数**:
- 响应时间: [具体数值，如 < 200ms]
- 资源占用: [具体数值，如 < 30MB 内存]
- [其他可量化指标]

**UI/UX 规范**（如有界面）:
- 窗口尺寸: [具体px值]
- 组件布局: [具体位置和间距]
- 视觉风格: [具体描述]
- 动画参数: [时长 + 缓动曲线]

**反面案例（NOT to do）**:
- [不要做什么，为什么]

**边界条件**:
- 异常情况1: [如何处理]
- 异常情况2: [如何处理]

## 3. 系统集成
- 权限需求: [具体权限列表]
- 系统API: [具体使用的系统API]
- 特殊行为: [如LSUIElement模式、后台运行等]

## 4. 工程化要求
- 构建方式: [具体构建工具和命令]
- 依赖管理: [具体的包管理器]
- 测试要求: [测试覆盖要求]
- 发布要求: [签名、打包、分发方式]
```

## 4. 工作流程

### 4.1 单次生成流程

1. **接收用户输入**：一句话或一段粗略描述
2. **意图分类**：识别平台类型、功能类型、复杂度级别
3. **知识检索**：从平台知识库中获取该类App的规范模板
4. **自动补全**：
   - 补充缺失的平台信息
   - 添加技术实现路径
   - 生成量化参数
   - 推断可能的边界情况
5. **PRD 组装**：按照标准模板格式化输出
6. **自检**：检查是否有未填充的占位符、矛盾点、歧义

### 4.2 自检机制

生成后的 PRD 必须通过以下检查：
- **占位符检查**：不能有 [TODO]、[待定]、[TBD] 等未填充项
- **量化检查**：所有参数必须包含具体数值
- **一致性检查**：各模块之间的技术栈必须一致
- **可执行性检查**：每个功能描述必须足够具体，能指导代码实现

## 5. 关键技术决策

### 5.1 实现形态

**推荐方案：作为 Claude Code Skill 实现**

理由：
- 用户已经在使用 Claude Code，投喂即用
- Skill 可以利用现有的文件读写、代码搜索等工具
- 可以输出到文件或直接显示
- 便于迭代和分享

**备选方案：作为独立 Agent 实现**
- 适合需要长时间、多轮次的生成任务
- 可以维护上下文状态
- 适合复杂的 PRD 生成

### 5.2 输出目标

生成的 PRD 目标："投喂给大模型后，大模型无需追问即可开始实现"。

检验标准：
- 大模型看到 PRD 后，是否还需要问"这个功能具体怎么实现？"
- 如果需要追问，则 PRD 不够详细

## 6. 使用场景

### 正面案例
- 用户输入："macOS菜单栏语音输入，按住Fn录音，松开注入文字"
  → 自动补全：Apple Speech Framework、流式识别、波形动画参数、CJK输入法处理、LLM纠错集成、LSUIElement模式

- 用户输入："做一个iOS拍照翻译App"
  → 自动补全：AVFoundation相机、Vision框架OCR、MLKit/Apple Neural Engine翻译、权限描述、UI布局

### 负面案例（Skill需要拒绝的场景）
- "帮我做一个像微信一样的App" → 太过模糊，需要先进行意图澄清
- "写一个深度学习框架" → 超出单次PRD的范围，需要拆解

---

## 7. 局限性 & 边界

### 7.1 Skill 能做到的
- macOS/iOS 桌面/移动App的详细PRD
- Web应用的全栈PRD（前端+后端+数据库）
- 命令行工具的PRD
- Chrome Extension的PRD
- 跨平台App（Flutter/React Native）的PRD

### 7.2 Skill 不能做到的
- 极度创新的产品（没有先例可供推断）
- 涉及硬件/嵌入式/操作系统的底层PRD
- 需要数学证明或算法创新的PRD
- 多产品线平台型PRD（需要先拆解）
