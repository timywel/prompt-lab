# 方案B设计文档：交互式对话型 PRD 构建器

## 状态
已接受

## 上下文
需要设计一个 PRD 生成器 agent/skill，通过多轮引导式对话探测用户需求，涵盖平台、功能、技术偏好、约束条件等维度，逐模块确认后聚合成完整 PRD。

---

## 1. 核心设计理念

### 1.1 交互式 vs 全自动的差异

| 维度 | 全自动补全型（方案A） | 交互式对话型（方案B） |
|------|---------------------|---------------------|
| **用户参与度** | 低——输入即输出 | 高——多轮问答 |
| **PRD 准确性** | 依赖知识库，可能有推断偏差 | 用户确认，准确度高 |
| **生成速度** | 快——秒级 | 慢——分钟级 |
|  **适用场景** | 常见类型App | 创新型/复杂型App |
| **知识库依赖** | 高 | 低（引导式挖掘） |
| **输出保证** | 可能遗漏关键需求 | 每个模块经过用户审核 |

### 1.2 核心原则

1. **每次只问一个问题**：避免信息过载
2. **选择题优先**：降低用户回答成本
3. **智能默认值**：每个问题都有推荐选项，用户可以直接回车跳过
4. **随时可打断**：用户可以随时说"就按这个来"
5. **进度可视化**：用户始终知道"还有几步"

## 2. 对话流程设计

### 2.1 总体流程

```
用户输入（任意形式：一句描述 / 一段文字 / 甚至一张草图描述）
    ↓
阶段0: 意图理解
    └─ 识别用户想要做的产品类型
    ↓
阶段1: 平台与基础（3-5个问题）
    ├─ 目标平台
    ├─ 目标用户
    ├─ 核心价值主张
    └─ MVP vs 完整版
    ↓
阶段2: 核心功能（3-8个问题）
    ├─ 主要交互方式
    ├─ 数据输入/输出
    ├─ 第三方集成
    └─ 离线能力需求
    ↓
阶段3: 技术偏好（2-4个问题）
    ├─ 技术栈偏好
    ├─ 团队技术栈（如果有）
    └─ 构建工具偏好
    ↓
阶段4: 质量要求（2-3个问题）
    ├─ 性能敏感度
    ├─ 兼容性要求
    └─ 测试覆盖要求
    ↓
PRD 生成 + 逐模块确认
    ↓
最终 PRD 输出
```

### 2.2 各阶段详细设计

#### 阶段0: 意图理解（1个问题）

**问题**: "看起来你想做一个 **[推断的平台]** **[推断的功能类型]**。请确认或补充："

**选项**:

- 修改为：______（用户自定义）

这个阶段利用方案A的意图识别能力，给出一个初始推断，用户可以确认或修正。

## 3. PRD 生成与确认

### 3.1 逐模块确认机制

对话收集完信息后，进入确认阶段：

```
PRD 已生成，包含以下模块：
[1] 平台与构建配置
[2] 核心功能模块A
[3] 核心功能模块B
[4] UI/UX 设计规范
[5] 系统集成需求
[6] 工程化要求
[7] 测试策略

请逐个确认或修改：
- 输入模块编号修改（如：2, 4）
- 输入 "通过" 继续生成完整 PRD
- 输入 "重新生成" 重新生成某个模块
- 输入 "跳过" 跳过确认直接输出
```

### 3.2 修改循环

用户修改某个模块 → agent 重新生成该模块 → 用户确认 → 下一个

最多3轮修改，3轮后给出警告："此模块已修改3次，建议人工审查最终效果。"

## 5. 关键技术决策

### 5.1 实现形态

**作为 Claude Code Skill + Agent 混合实现**

- **Skill 部分**：定义对话流程、问题模板、选项数据
- **Agent 部分**：管理对话状态、跟踪已回答的问题、动态生成后续问题

### 5.2 状态管理

由于对话可能跨越多个消息，需要维护状态：

```yaml
对话状态:
  current_phase: 2  # 当前阶段
  current_question: 7  # 当前问题编号
  answered_questions: [1, 2, 3, 4, 5, 6]  # 已回答
  user_inputs:  # 用户的所有回答
    platform: "macOS"
    min_version: "macOS 14+"
    interaction: "语音驱动"
    ...
  generated_modules: []  # 已生成的PRD模块
  pending_confirmations: []  # 待确认的模块
```

### 5.3 智能推断

利用方案A的知识库，在用户回答问题的同时，agent 可以进行推断：
- 如果用户回答"语音驱动"，自动标记需要：麦克风权限、Speech Framework、波形UI、可能的LLM集成
- 这些推断不需要用户回答，但会在PRD中体现

## 6. 使用场景

### 正面案例
- 用户想做"一个帮助视障人士的iOS导航App" → 交互式引导出无障碍需求、VoiceOver集成、精确导航API、离线地图等关键需求
- 用户想做"团队内部的代码片段分享工具" → 引导出权限管理、markdown渲染、搜索需求、可能的后端服务

### 需拒绝的场景
- 用户输入过于具体、已经是一个完整PRD → 建议改用方案A的全自动补全
- 用户想要的是一整个平台（如"做一个像抖音一样的平台"）→ 建议拆解为多个子PRD

## 7. 与方案A的协作

方案B可以调用方案A作为"输出引擎"：
- 对话收集信息 → 方案A的知识库补全细节 → 方案B的确认流程 → 最终输出
