# PRD-Orchestrator: PRD 生成协调层

## 定位

PRD-Orchestrator 是 PRD 生成的**统一入口**。用户只需说"帮我生成PRD"，它自动：
1. 分析输入复杂度
2. 智能路由到最合适的方案组合
3. 串联执行
4. 进行质检

## 激活条件

当用户说以下内容时激活：
- "帮我生成PRD"
- "写一个PRD"
- "帮我写PRD"
- "生成PRD"
- 包含 "prd-orchestrator" 触发词

激活后，此 Skill 成为协调层，自动判断使用哪个下游方案。

---

## 复杂度分析

激活后首先分析用户输入的复杂度，将输入分为四类：

| 复杂度 | 特征 | 路由 |
|--------|------|------|
| **简单型** | 一句话描述，单一功能，常见平台 | 直接 → 方案A |
| **已有PRD型** | 用户提供了初步PRD文本或文件路径 | 直接 → 方案C |
| **复杂/创新型** | 多功能、特殊平台、创新交互、新用户类型 | 方案B → 方案A → 方案C |
| **不确定型** | 描述模糊或功能不明确 | 方案B 需求挖掘 |

### 复杂度分析（生成前评估）

激活后，首先调用 `prd-qa` 的生成前评估模块：

1. 读取用户输入
2. 调用 PRD-QA 的"生成前评估"流程
3. 展示评估报告（包含：复杂度等级、知识库覆盖度、冲突检测、生成建议）
4. 询问用户："按推荐方案生成？" / "我选另一个方案" / "修改需求"
5. 用户确认后，按推荐方案路由

---

## 智能路由流程

```
用户输入
  ↓
复杂度评估（展示评估结果）
  ↓
┌─ 简单型 ──────────────────→ 方案A ──────────────────→ 方案C（可选）
├─ 已有PRD型 ───────────────→ 方案C
├─ 复杂/创新型 ───→ 方案B ─→ 方案A ─→ 方案C ─→ 方案B二轮确认
└─ 不确定型 ──────→ 方案B 需求挖掘 ─→ 重新评估
  ↓
[动态增强] → 自动注入缺失模板
  ↓
PRD-QA 质检
  ↓
输出 + 质量报告
```

### 路由到下游方案的映射

- **方案A**：prd-autofill（全自动 PRD 生成器）
- **方案B**：prd-conversational（交互式 PRD 构建器）
- **方案C**：prd-deep-expand（深度扩展型 PRD 生成器）
- **PRD-QA**：使用方案C 中的质检逻辑对最终输出进行质量检查

## 串联执行规范

### 路径一：简单型路径

适用场景：输入长度 < 100字，单一功能，常见平台。

执行步骤：

```
[1/5] 调用方案A（prd-autofill）生成基础PRD
      ↓
[2/5] 询问用户："是否需要深度扩展？"
      ├─ 用户同意 → [3/5] 调用方案C（prd-deep-expand）
      └─ 用户拒绝 → 跳过 [3/5]
      ↓
[3/5]（若用户同意）方案C 深度扩展
      ↓
[4/5] [动态增强] → 自动注入缺失模板
      ↓
[5/5] PRD-QA 质检
      ↓
输出最终结果
```

### 路径二：复杂/创新型路径

适用场景：输入长度 > 200字，多功能，特殊平台或创新交互。

执行步骤：

```
[1/8] 方案B 第一轮：意图 + 平台 + 核心功能
      ↓
[2/8] 方案B 第二轮：细节 + 技术偏好 + 质量
      ↓
[3/8] 方案B 第三轮：高级特性 + AI服务 + 交付格式
      ↓
[4/8] 基于方案B输出 → 方案A 补充技术细节
      ↓
[5/8] 基于方案A输出 → 方案C 深度扩展
      ↓
[6/8] 方案B 二轮确认：最终评审
      ↓
[7/8] [动态增强] → 自动注入缺失模板
      ↓
[8/8] PRD-QA 质检
      ↓
输出最终结果
```

### 路径三：已有PRD型路径

适用场景：用户提供了初步PRD文本（Markdown块）或文件路径。

执行步骤：

```
[1/4] 解析用户提供的PRD内容
      ↓
[2/4] 方案C（prd-deep-expand）直接扩展
      ↓
[3/4] [动态增强] → 自动注入缺失模板
      ↓
[4/4] PRD-QA 质检
      ↓
输出最终结果
```

### 路径四：不确定型路径

适用场景：输入模糊，功能不明确，无法做出可靠评估。

执行步骤：

```
[1/N] 方案B 需求挖掘（交互式引导）
      ↓
[2/N] 基于挖掘结果重新评估复杂度
      ↓
[3/N] 根据新的评估结果走对应路径
      ↓
[.../N] [动态增强] → 自动注入缺失模板
      ↓
[N/N] PRD-QA 质检
      ↓
输出最终结果
```

## 协调层职责

### 1. 上下文传递

确保方案之间的上下文连贯传递：

- **方案B → 方案A**：将方案B输出的用户确认需求，作为方案A的知识库上下文
- **方案A → 方案C**：将方案A输出的技术细节补全的PRD，作为方案C的初步PRD
- **各方案 → PRD-QA**：将最终输出传递给质检模块

上下文传递时，使用简洁的摘要格式，不重复完整内容。

### 2. 用户决策点

每个路径的分支点必须询问用户，不自动决策：

| 决策点 | 询问内容 |
|--------|----------|
| 复杂度评估后 | "是否同意此路径？" |
| 方案A完成后 | "是否需要深度扩展？" |
| 复杂路径中途 | "是否继续下一步？" |
| 最终结果后 | "是否需要修改某个章节？" |

### 3. 进度展示

每个步骤完成后展示当前进度，使用统一格式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 PRD 生成中...

[1/5] ✅ 方案B 需求挖掘完成
[2/5] 🔄 方案A 技术补全中...
[3/5] ⏳ 方案C 深度扩展
[4/5] ⏳ 动态增强
[5/5] ⏳ PRD-QA 质检

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

进度符号说明：
- ✅ 已完成
- 🔄 进行中
- ⏳ 等待执行
- ❌ 已跳过

### 4. 可中断性

用户可随时说"停"、"够了"、"中止"终止后续步骤。收到中断指令后：
1. 立即停止执行当前步骤
2. 基于已完成的内容输出当前结果
3. 调用 PRD-QA 对已有内容进行质检
4. 输出当前质量报告

### 5. 质量担保

无论用户选择哪个路径，最终输出必须经过 PRD-QA 质检。

## 动态增强规则

在任意方案（A/B/C）生成 PRD 后、输出前，自动执行以下增强注入：

### 自动注入检查清单

每个方案生成后，逐一检查以下内容是否已存在于 PRD 中：

#### 1. Info.plist / Entitlements 模板（权重：高）
```
检测: 搜索 "NSMicrophoneUsageDescription" 或 "Privacy"
├── 已存在 → ✅ 跳过
└── 不存在 → 注入: `_shared/platform-configs/[platform]-infoplist.yaml`
         → 注入: `_shared/platform-configs/[platform]-entitlements.yaml`
```

#### 2. 测试金字塔章节（权重：高）
```
检测: 搜索 "测试金字塔" 或 "E2E" 或 "Unit"
├── 已存在 → ✅ 跳过
└── 不存在 → 注入: `_shared/test-templates/test-pyramid-template.md` 的简化版
```

#### 3. YAML 组件规范（权重：中）
```
检测: 搜索 "type:" 和 "size:" 和 "layout:"
├── 已存在（YAML格式）→ ✅ 跳过
└── 不存在（文字描述）→ 注入简化YAML组件规范模板
```

#### 4. 自检清单引用（权重：中）
```
检测: 搜索 "self-review" 或 "自检" 或 "检查清单"
├── 已存在 → ✅ 跳过
└── 不存在 → 在末尾追加: "补充：完成本PRD后，请参考 ../_shared/qa-checks/self-review-checklist.md 进行14项自检"
```

#### 5. 动画时长一致性检查（权重：高）
```
检测: 搜索所有动画时长数值（单位：ms 或 s）
├── 无重复 → ✅ 跳过
└── 存在矛盾（如300ms vs 500ms vs 350ms）→ 统一为350ms
```

#### 6. 量化参数完整性检查（权重：中）
```
检测: 搜索性能指标（延迟/内存/CPU/帧率）
├── 每个核心功能都有 → ✅ 跳过
└── 存在缺失 → 自动填充默认值（引用量化参数生成器的标准值）
```

### 动态增强执行流程

```
PRD生成完成
  ↓
[自动增强检查]
├── 1. Info.plist → 未找到 → 注入模板
├── 2. 测试金字塔 → 未找到 → 注入模板
├── 3. YAML规范 → 文字描述 → 转换为简化YAML
├── 4. 自检引用 → 未找到 → 追加引用
├── 5. 动画时长 → 发现矛盾 → 统一为350ms
└── 6. 量化参数 → 发现缺失 → 填充默认值
  ↓
增强后的PRD
  ↓
输出 + 告知用户："已自动注入X项增强"
```

## 与下游方案的配合

### 方案A（prd-autofill）

- **触发方式**：通过 Skill 指令 "prd-autofill" 调用
- **输入**：用户的简单需求描述或方案B的需求摘要
- **输出**：自动补全的完整 PRD
- **适用场景**：快速生成，不需要多轮交互

### 方案B（prd-conversational）

- **触发方式**：通过 Skill 指令 "prd-conversational" 调用
- **输入**：用户模糊或复杂的需求
- **输出**：经过多轮引导确认的完整需求文档
- **适用场景**：需求不明确，需要深度挖掘

### 方案C（prd-deep-expand）

- **触发方式**：通过 Skill 指令 "prd-deep-expand" 调用
- **输入**：已有的初步PRD
- **输出**：从架构/UI/UX/工程化/测试/边界/运维六个维度深度扩展的PRD
- **适用场景**：已有基础PRD，需要全面深化

---

## 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 下游方案调用失败 | 重试一次，失败则询问用户是否跳过该方案 |
| 用户中断 | 立即停止，输出已完成内容并质检 |
| 输入完全不明确 | 强制走方案B路径进行需求挖掘 |
| 方案间上下文传递失败 | 重新传递或要求用户复述关键信息 |
